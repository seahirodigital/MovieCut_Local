# 動画内文字検出 Apple Vision OCR 検証計画

## 目的

`/Users/user/Library/CloudStorage/OneDrive-個人用/開発/Movie_AutoCut/MovieCut_Local` に保存されている動画を対象に、動画フレーム内へ韓国語テキスト `벗어` が表示されているかを検出し、検出された動画を「採用動画」として移動できるかを検証する。

今回の目的は実装そのものではなく、実装可能性・安全な運用方法・最小検証手順を整理すること。

## 検証したいこと

1. 動画内のフレームから `벗어` を OCR で検出できるか。
2. Apple Vision OCR が韓国語文字を安定して認識できるか。
3. どの程度のフレーム抽出間隔なら、速度と検出精度のバランスがよいか。
4. 検出結果をもとに、動画を削除ではなく安全に「採用動画」フォルダへ移動できるか。
5. 誤検出・見逃しが発生した場合に、人間が確認しやすいログや一覧を残せるか。

## 推奨方針

最初は Mac 専用機能として Apple Vision Framework の OCR を使う。

理由:

- `/Users/user/Library/CloudStorage/OneDrive-個人用/開発/Movie_AutoCut/MovieCut_Local/Mac/server_mac.py` が既に Mac 専用処理の置き場になっているため、Apple Vision を利用する構成と相性がよい。
- Tesseract OCR や EasyOCR よりも、Mac 環境では追加依存を軽くしやすい。
- OCR 検証の初期段階では、ローカルで完結し、動画ファイルを外部サービスへ送らない構成の方が安全。
- `벗어` のような特定ワード検出であれば、「全文の完全認識」よりも「候補文字列に対象ワードが含まれるか」を確認するだけでよい。

## 全体構成案

```text
/Users/user/Library/CloudStorage/OneDrive-個人用/開発/Movie_AutoCut/MovieCut_Local/Mac/server_mac.py
  ↓
動画パスを指定して OCR 検証を開始
  ↓
ffmpeg または OpenCV で動画から数フレームを抽出
  ↓
Apple Vision OCR で各フレームの文字を認識
  ↓
認識テキストに「벗어」が含まれるか判定
  ↓
検出結果を JSON またはログとして保存
  ↓
検出された動画を採用動画フォルダへ移動する候補にする
```

## 重要な安全方針

最初から動画を削除しない。

理由:

- OCR は誤検出する可能性がある。
- 背景の看板、字幕、UI、ロゴ、別言語の類似文字を誤って対象ワードとして読む可能性がある。
- 動画削除は取り返しがつきにくいため、初期検証では「移動」または「候補一覧化」に留めるのが安全。

推奨する初期運用:

```text
検出なし
  → 何もしない

「벗어」検出あり
  → 採用動画候補フォルダへ移動
  → 人間がレビュー
  → 問題なければ正式採用
```

## 保存先フォルダ案

採用動画の移動先は、初期検証では次のような専用フォルダを使う。

```text
/Users/user/Library/CloudStorage/OneDrive-個人用/開発/Movie_AutoCut/MovieCut_Local/採用動画_벗어検出/
```

理由:

- 既存の OK 移動先や削除移動先と混ぜると、検証結果と通常レビュー結果の区別がつきにくくなる。
- 専用フォルダに分けることで、誤検出があっても元の運用へ影響しにくい。
- 後から検出精度を確認しやすい。

## フレーム抽出方針

動画全フレームに OCR をかけるのは避ける。

理由:

- 30fps の動画では 1 分で約 1800 フレームある。
- 全フレーム OCR は処理時間が長く、レビュー作業のテンポを大きく落とす。
- 検証目的が「対象ワードが出る動画を拾うこと」であれば、まずは少数フレームで十分に傾向を見られる。

初期検証の推奨モード:

```text
高速モード:
  先頭 10%
  中央 50%
  終盤 90%
  合計 3 フレーム

標準モード:
  3 秒ごとに 1 フレーム
  最大 20 フレーム

精密モード:
  1 秒ごとに 1 フレーム
  最大 60 フレーム
```

最初は標準モードを推奨する。

理由:

- 3 フレームだけでは、対象文字が一瞬だけ出る動画を見逃す可能性がある。
- 1 秒ごとは精度が上がるが、検証初期には重すぎる可能性がある。
- 3 秒ごと最大 20 フレームなら、速度と検出機会のバランスがよい。

## OCR 判定条件案

対象ワード:

```text
벗어
```

基本判定:

```text
OCR 結果のいずれかに「벗어」が含まれていれば検出あり
```

補助判定:

```text
Apple Vision OCR の信頼度が取得できる場合:
  confidence >= 0.6 を優先

信頼度が低い場合:
  移動せず「要確認候補」としてログだけ残す
```

誤認識対策:

- OCR 結果の空白を除去してから比較する。
- 改行をまたいで `벗어` が出る可能性を考慮する。
- 韓国語 OCR の揺れがある場合は、検出テキスト全文をログに残して実例を確認する。

## ログ出力案

検証段階では、動画を移動する前に必ずログを残す。

ログ保存先案:

```text
/Users/user/Library/CloudStorage/OneDrive-個人用/開発/Movie_AutoCut/MovieCut_Local/Mac/movie vision/ocr_detection_log.json
```

ログ項目:

```json
{
  "video_path": "/Users/user/Library/CloudStorage/OneDrive-個人用/開発/Movie_AutoCut/MovieCut_Local/input/sample.mp4",
  "target_word": "벗어",
  "detected": true,
  "matched_text": "벗어",
  "frame_time_sec": 12.0,
  "confidence": 0.82,
  "action": "move_candidate",
  "destination_path": "/Users/user/Library/CloudStorage/OneDrive-個人用/開発/Movie_AutoCut/MovieCut_Local/採用動画_벗어検出/sample.mp4"

```

理由:

- 誤検出した場合に、どの動画の何秒付近で判定されたかを追える。
- 後から判定条件を調整する材料になる。
- 自動移動した根拠を確認できる。

## 段階的な検証手順

### 1. 単体動画で OCR 検証

1 本の動画を指定し、数フレームだけ OCR する。

理由:

- Apple Vision OCR が `벗어` を認識できるかを最短で確認するため。
- 先に精度と速度の癖を見てから、移動処理へ進む方が安全。

成功条件:

- OCR 結果に認識テキストが出る。
- `벗어` を含むフレームで検出ありになる。
- 検出時刻と検出テキストがログに残る。

### 2. 複数動画で検出のみ実行

複数の動画に対して、移動せず検出ログだけ作成する。

理由:

- いきなりファイルを動かすと、誤検出時の確認が面倒になる。
- まずログだけ見れば、判定条件が厳しすぎるか緩すぎるか判断できる。

成功条件:

- 検出あり動画と検出なし動画が一覧化される。
- 誤検出の傾向が確認できる。
- 処理時間が許容範囲に収まる。

### 3. 採用動画候補フォルダへ移動

検出ありの動画だけを、専用の採用動画候補フォルダへ移動する。

理由:

- 削除ではなく移動にすることで、誤検出しても戻せる。
- 既存レビュー機能へ組み込む前に、単独機能として安全に検証できる。

成功条件:

- 検出あり動画だけが `/Users/user/Library/CloudStorage/OneDrive-個人用/開発/Movie_AutoCut/MovieCut_Local/採用動画_벗어検出/` へ移動される。
- 元ファイル名が維持される。
- 同名ファイルがある場合は上書きせず、連番または日時付きファイル名で保存される。

### 4. レビュー画面への統合を検討

検証結果が安定してから、レビュー画面に次の機能を追加する。

- `벗어` 検出ありタグ
- `벗어` 検出ありだけ表示
- `벗어` 検出ありを採用動画へ移動

理由:

- OCR 精度が不明な段階で UI を作り込むと、後で仕様変更が大きくなる。
- 先にログ検証で判定条件を固めた方が、UI がシンプルになる。

## 実装時の候補 API

将来的に実装する場合は、次のような API が考えられる。

```text
POST /api/mac/detect-text-in-video
```

入力:

```json
{
  "video_path": "/Users/user/Library/CloudStorage/OneDrive-個人用/開発/Movie_AutoCut/MovieCut_Local/input/sample.mp4",
  "target_word": "벗어",
  "sample_mode": "standard",
  "move_on_detect": false
}
```

出力:

```json
{
  "success": true,
  "detected": true,
  "target_word": "벗어",
  "matches": [
    {
      "frame_time_sec": 12.0,
      "text": "벗어",
      "confidence": 0.82
    }
  ],
  "action": "none"
}
```

## 実装時の注意点

- 日本語・韓国語を含むログや設定ファイルは UTF-8 で保存する。
- 動画パスは必ず `/Users/user/Library/CloudStorage/OneDrive-個人用/開発/Movie_AutoCut/MovieCut_Local/...` のような完全フルパスで扱う。
- 移動処理では上書きを禁止する。
- OCR 処理中にアプリ全体が固まらないように、バックグラウンド処理または進捗表示を用意する。
- 最初は削除処理と接続しない。
- 検出ログには、検出文字列・時刻・信頼度・移動先を残す。

## リスクと対策

| リスク | 内容 | 対策 |
| --- | --- | --- |
| 誤検出 | `벗어` ではない文字を対象ワードとして読む | 信頼度しきい値、ログ確認、移動前確認を入れる |
| 見逃し | 対象文字が出たフレームをサンプリングできない | 標準モードや精密モードを選べるようにする |
| 処理が重い | 多数動画に OCR をかけると時間がかかる | 最大フレーム数を制限し、進捗表示を出す |
| ファイル消失 | 自動削除や上書きで動画を失う | 初期段階では削除禁止、移動先の上書き禁止 |
| 文字化け | 韓国語を含むログが壊れる | UTF-8 で保存し、Bash の echo / heredoc で日本語・韓国語を書き込まない |

## 最初の検証完了条件

次の状態になれば、第一段階の検証は成功とする。

1. `벗어` を含む動画で検出ありになる。
2. `벗어` を含まない動画で検出なしになる。
3. 検出ログに対象動画、検出時刻、検出テキスト、信頼度が残る。
4. 自動削除は行わず、必要に応じて採用動画候補フォルダへ移動できる。
5. 誤検出した場合でも、元動画を失わずに確認・復旧できる。

## 結論

この機能は実装可能。

最初の実装候補としては、Mac 専用の Apple Vision OCR を使い、動画から少数フレームを抽出して `벗어` を検出する方式が最も現実的。初期段階では削除ではなく、`/Users/user/Library/CloudStorage/OneDrive-個人用/開発/Movie_AutoCut/MovieCut_Local/採用動画_벗어検出/` への移動候補化に留めるのが安全。

まずは「単体動画で検出ログを出す」小さな検証から始め、精度と速度を確認してからレビュー画面や自動移動へ広げるのがよい。
